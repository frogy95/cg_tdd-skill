# /spec — 명세 작성 스킬

## 역할
기능의 **의도(Why)**, **목표(What)**, **제약 조건**, **엣지 케이스**를 정의한다.
구현 방법(How)은 이 단계에서 다루지 않는다.

## 규칙
- **코드를 작성하거나 수정하지 않는다**
- 불명확한 점이 있으면 구현 대신 질문한다
- 산출물은 `specs/<feature-name>.md` 파일 하나뿐이다
- 명세는 테스트 케이스를 도출할 수 있을 만큼 구체적이어야 한다

## 명세 문서 구조

`specs/<feature-name>.md`에 다음 구조로 작성한다:

```markdown
# [기능명] 명세

## Why (왜 필요한가)
이 기능이 필요한 이유와 해결하는 문제

## What (무엇을 하는가)
기능의 목표와 기대 동작

## 제약 조건
- 반드시 지켜야 할 기술적/비즈니스 제약
- 성능 요구사항, 보안 요구사항 등

## 입력 / 출력
- 입력: 어떤 데이터/이벤트를 받는가
- 출력: 어떤 결과/부작용을 만드는가

## 엣지 케이스
- 경계값, 예외 상황, 실패 시나리오

## 완료 기준 (Acceptance Criteria)
- [ ] 기준 1
- [ ] 기준 2
```

## 실행 절차

1. 사용자의 요청에서 **기능 이름**과 **맥락**을 파악한다
2. 불명확한 부분이 있으면 **질문으로 명확화**한다 (코드 작성 금지)
3. `specs/<feature-name>.md` 파일을 생성한다
4. 명세 작성 완료 후 **다음 단계 안내**: `/red`로 실패 테스트를 작성하세요

## 완료 신호
명세 파일 생성 후 다음을 출력한다:
```
✅ 명세 완료: specs/<feature-name>.md
다음 단계: /red — 실패하는 테스트를 작성합니다
```
