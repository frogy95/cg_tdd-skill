# /commit — 커밋 스킬

## 역할
변경 사항을 분석하여 **의도와 맥락을 담은 한국어 커밋 메시지**를 생성하고 커밋한다.
"무엇을 했는가"가 아닌 **"왜 했는가"**를 메시지에 담는다.

## 규칙
- 커밋 전 반드시 `/verify`가 통과되어야 한다
- 커밋 메시지는 **한국어**로 작성한다
- `git add .` 대신 변경된 파일을 명시적으로 스테이징한다
- 민감한 파일(`.env`, 인증 정보)은 커밋하지 않는다

## 커밋 메시지 형식

```
<타입>: <제목> (50자 이내)

<본문> (선택사항 — 왜 이렇게 했는지 설명)
```

### 타입 목록
| 타입 | 용도 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없는 코드 개선 |
| `test` | 테스트 추가/수정 |
| `docs` | 문서 변경 |
| `chore` | 빌드, 설정 변경 |

### 좋은 커밋 메시지 예시
```
feat: 이메일 중복 검사 기능 추가

회원가입 시 이미 가입된 이메일은 즉시 오류를 반환하도록 함.
기존에는 DB INSERT 실패 후에야 중복을 알 수 있었음.
```

## 실행 절차

1. `git status`로 변경된 파일 목록을 확인한다
2. `git diff`로 변경 내용을 파악한다
3. 변경의 **의도와 맥락**을 분석한다
4. 적절한 타입과 제목으로 커밋 메시지를 작성한다
5. 관련 파일만 스테이징하고 커밋한다

## 완료 신호
커밋 성공 후:
```
📦 커밋 완료
메시지: <커밋 메시지>
해시: <짧은 해시>

🎉 파이프라인 완료: /spec → /red → /green → /refactor → /verify → /commit
```
