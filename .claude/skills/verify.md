# /verify — 검증 스킬

## 역할
**하네스 엔지니어링**의 핵심: 결정적 도구로 AI 산출물을 객관적으로 검증한다.
인간의 판단이 아닌 **도구의 판단**으로 품질을 확인한다.

## 왜 검증 단계가 필요한가?

> "AI가 작성한 코드는 그럴듯해 보이지만 틀릴 수 있다.
> 컴파일러와 테스트는 거짓말하지 않는다."

검증 게이트는 AI의 환각(hallucination)과 실수를 잡아내는 안전망이다.

## 검증 항목

프로젝트에 존재하는 도구를 순서대로 실행한다:

### 1. 테스트
```bash
# 프로젝트에 맞는 명령 실행
npm test / pytest / go test ./... / cargo test
```

### 2. 타입 체크 (해당되는 경우)
```bash
npx tsc --noEmit / mypy . / pyright
```

### 3. 린터
```bash
npm run lint / flake8 / golangci-lint run
```

### 4. 빌드 (해당되는 경우)
```bash
npm run build / go build ./... / cargo build
```

## 실행 절차

1. 프로젝트에서 사용 가능한 도구를 파악한다 (`package.json`, `Makefile` 등 확인)
2. 각 도구를 순서대로 실행한다
3. 결과를 수집하여 리포트를 작성한다
4. 실패가 있으면 원인을 분석하고 해당 단계로 되돌아간다

## 결과 리포트 형식

```
🔍 Verify 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━
테스트:     ✅ N개 통과 / ❌ N개 실패
타입 체크:  ✅ 오류 없음 / ❌ N개 오류
린터:       ✅ 경고 없음 / ⚠️ N개 경고
빌드:       ✅ 성공 / ❌ 실패
━━━━━━━━━━━━━━━━━━━━━━━━━━
종합: ✅ 검증 통과 → /commit 진행 가능
      ❌ 검증 실패 → [해당 단계]로 돌아가세요
```

## 실패 시 처리
| 실패 유형 | 돌아갈 단계 |
|-----------|-------------|
| 테스트 실패 | `/green` (구현 수정) |
| 타입 오류 | `/green` 또는 `/refactor` |
| 린터 경고 | `/refactor` |
| 빌드 실패 | `/green` |
